## STEP 1 — SA (Blueprint)

---

> You are the Study Systems Analyst. Read your specialist file at `study/agents/STUDY_SYSTEMS_ANALYST.md` first. Read the diagnostic at `study/knowledge/research/clean-routes-diagnostic-2026-03-22.md` — it maps every `setScreen()` call site (40+ transitions), all hierarchy violations, and the current back mechanism. **Task:** Blueprint a clean navigation system with strict hierarchy. **CEO decisions:** (1) Navigation stack with unlimited depth — back always pops to previous screen. (2) No lateral jumps between course options — all lateral navigation removed, user must go through courseHome. (3) Profile keeps "Practice This Skill" but returns to Profile after session, not course hierarchy. **Design these changes:** (1) **Navigation stack.** Replace `previousScreen` (single value) with `navStack` (array). Define a `navigateTo(screen)` function that pushes current screen onto the stack before setting the new screen. Define a `goBack()` function that pops the stack and sets screen to the popped value (fallback to `home` if stack is empty). Every `setScreen()` call in the app must route through `navigateTo()` or `goBack()` — no direct `setScreen()` except during initialization. Specify the state declaration, both functions, and the context value updates. (2) **Remove lateral jumps.** The diagnostic lists these violations: CurriculumScreen→materials (lines 206, 492), CurriculumScreen→schedule (line 488), ScheduleScreen→curriculum (line 394), ManageScreen→materials (line 35), ManageScreen→skills (line 46). For each, specify the replacement behavior — either remove the button entirely or change it to `goBack()` (back to courseHome, where the user can navigate to the target). Prefer removing navigation buttons and showing a text hint instead (e.g., "Go to Course Home to access Materials"). (3) **TopBarButtons fix.** Profile and Notifications buttons currently call `setScreen("profile")` / `setScreen("notifs")` directly. Change to `navigateTo("profile")` / `navigateTo("notifs")`. Their back buttons change from hard-coded `setScreen("home")` to `goBack()`. This makes them inward-only — they always return to where the user came from. (4) **Profile study session return.** Profile's "Practice This Skill" calls `enterStudy()`. The `enterStudy()` function currently sets `previousScreen`. With the stack, `enterStudy` should call `navigateTo("study")` which pushes Profile onto the stack. When the study session ends, `goBack()` pops back to Profile. Verify this works with the existing `enterStudy` and `bootWithFocus` flows. (5) **System-initiated navigation.** The diagnostic found 3 system-initiated transitions: addMats→materials (StudyContext:692), ExtractionProgress click→materials, course deletion→home. These should use `navigateTo()` not `setScreen()`, so the user can go back from them. (6) **Upload screen hierarchy.** Currently `upload` is accessible from `home`. Should it be accessible from `courseHome` too? Or only from `home`? Specify where it fits in the hierarchy. **Blueprint must include:** the `navStack` state declaration, `navigateTo()` and `goBack()` function implementations, a complete migration checklist of every `setScreen()` call site (from the diagnostic's 40+ transitions) mapped to `navigateTo()` or `goBack()`, and a "How to verify" section. **Deposit:** `study/knowledge/architecture/clean-routes-blueprint-2026-03-22.md`. Next is Step 2 (DEV). Stop here and wait for user confirmation before proceeding.
