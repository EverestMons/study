## STEP 1 — SA (Blueprint)

---

> You are the Study Systems Analyst. Read your specialist file at `study/agents/STUDY_SYSTEMS_ANALYST.md` first. Read the diagnostic at `study/knowledge/research/chunk-relationships-diagnostic-2026-03-22.md` — it maps the full chunk data model, all existing relationships, and all missing capabilities. Then read `src/lib/minhash.js` (MinHash implementation), `src/lib/study.js` (focus on `loadFacetBasedContent` and the context assembly sections), and `src-tauri/migrations/` to see the current migration count. **Task:** Blueprint 4 chunk relationship improvements. **Change 1 — Persist MinHash similarity pairs.** New migration (009): create `chunk_similarities` table: `chunk_a_id TEXT NOT NULL REFERENCES chunks(id) ON DELETE CASCADE, chunk_b_id TEXT NOT NULL REFERENCES chunks(id) ON DELETE CASCADE, similarity REAL NOT NULL, UNIQUE(chunk_a_id, chunk_b_id)` with canonical ordering (a < b). In `minhash.js`, after `findNearDuplicates()` returns results, store each pair in the new table via a new `ChunkSimilarities.createBatch(pairs)` DB method. Only store pairs above a threshold (0.5 — lower than the dedup threshold of 0.7 to capture "related but not duplicate"). Specify the exact insertion point in the extraction pipeline where this runs. **Change 2 — Parse section_path into queryable tree.** The `section_path` column already stores paths like "Chapter 5 > Section 5.1". Add a utility function `parseSectionPath(path)` that returns `{ parts: ["Chapter 5", "Section 5.1"], depth: 2, parent: "Chapter 5", isRoot: false }`. Add a `getChunkTree(materialId)` function to `db.js` that loads all chunks for a material and organizes them into a tree structure: `{ label, chunkId, children: [...] }` where children are derived from section_path parsing. Specify how parent-child relationships are determined from section_path strings. **Change 3 — Expose ordering + section_path to tutor context.** In `study.js`, when chunks are assembled for the AI context, currently they appear as `--- [label] ---\n[content]`. Change to include position metadata: `--- [label] (Section: [section_path], Position: [ordering] of [total]) ---\n[content]`. Also add a DOCUMENT STRUCTURE block at the top of the chunk context section listing the material's section tree (from Change 2) as an outline — this gives the tutor a table-of-contents view. Keep it compact — just the tree labels, no content. **Change 4 — Chunk-level prerequisite ordering.** This is the most complex change. After extraction, when skills/facets are bound to chunks, infer prerequisite ordering between chunks: (a) **Document order heuristic:** within the same material, if chunk A has `ordering` < chunk B and they share facets under the same parent skill, chunk A is likely prerequisite to chunk B. (b) **Skill prerequisite transitivity:** if facet P on chunk A has a `prerequisite` type facet_concept_link to facet Q on chunk B, then chunk A is prerequisite to chunk B. Store in a new table `chunk_prerequisites(chunk_id TEXT, prerequisite_chunk_id TEXT, source TEXT, UNIQUE)` where `source` is "document_order" or "skill_link". The context builder can use this to order chunks and tell the tutor "this builds on [previous chunk label]". **Blueprint must include:** migration 009 SQL (both tables), all function signatures, the section_path parsing algorithm, the context format changes with examples, the prerequisite inference algorithm, and a "How to verify" section. **Deposit:** `study/knowledge/architecture/chunk-relationships-blueprint-2026-03-22.md`. Next is Step 2 (DEV). Stop here and wait for user confirmation before proceeding.

## STEP 4 — DEV (Prerequisite ordering)

---

> You are the Study Developer. Read your specialist file at `study/agents/STUDY_DEVELOPER.md` first. Read the SA blueprint at `study/knowledge/architecture/chunk-relationships-blueprint-2026-03-22.md` — implement prerequisite ordering. **Task 1 — Prerequisite inference.** Create `inferChunkPrerequisites(materialId, courseId)` function. Two inference methods per blueprint: (a) **Document order heuristic:** load all chunks for the material ordered by `ordering`. For each pair of chunks that share facets under the same parent skill (via `chunk_facet_bindings` → `facets` → `parent_skill_id`), if chunk A comes before chunk B, create a prerequisite record with source "document_order". Only create links between chunks that share a parent skill — not between every sequential pair (that would be noise). (b) **Skill prerequisite transitivity:** query `facet_concept_links` where `link_type = 'prerequisite'`. For each link, find chunks bound to each facet via `chunk_facet_bindings`. Create prerequisite records with source "skill_link". Store all results via `ChunkPrerequisites.create()`. **Task 2 — Hook into extraction pipeline.** Call `inferChunkPrerequisites` after the existing `wireCrossChapterPrereqs` step in the extraction flow (skills.js). Wrap in try/catch — prerequisite inference failure should not block extraction. **Task 3 — Context builder integration.** When the context builder loads chunks for a study session, query `ChunkPrerequisites.getByChunk(chunkId)` for the focused chunks. If a chunk has prerequisites, add a note in its context header: `--- [label] (Section: [path], Builds on: [prereq label]) ---`. This tells the tutor which concepts the student should already know. **Constraints:** Do not change FSRS. The document order heuristic should be conservative — only link chunks that share parent skills, not all sequential chunks. Limit to same-material prerequisites (cross-material prereqs are handled by facet concept links). The prerequisite table uses INSERT OR IGNORE for idempotency. Build verify. Commit with message: `"feat: chunk prerequisite ordering — document order + skill link inference"`. Standard prompt feedback protocol → `study/knowledge/research/agent-prompt-feedback.md`.
